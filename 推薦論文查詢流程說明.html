1. 系統概觀：兩張表的分工

整個系統由兩張表構成，一張紀錄「誰想要」，一張紀錄「PQ 上的真實資料」。

📥 recommends

需求／請求紀錄
  • 每筆使用者推薦就新增一列
  • 同一 DOI 多次推薦 → 累加 times
  • 書目欄位（title…）一開始是空的，查到後回填
  • 帶有 ck_flag、valid 狀態

📚 recommends_on_pq

ProQuest 來源真實資料（快取）
  • 一個 DOI 一列（PQ 上的權威資料）
  • 儲存 pq_id、title、author、year、school…
  • 是「真相表」，供 recommends 回填與比對
  • 查到後下次直接取用，不必再呼叫 PQ API

2. 完整資料流

使用者何時進入推薦？（觸發動線）

推薦的兩種來源：主動 vs 被動（active 欄位）

🙋 主動推薦（active=1）

RecommendController@store · 推薦表單
  • 使用者主動填寫推薦表單（姓名／系所／研究主題／Email）
  • 針對已顯示、已知的論文（表單由 recommends_on_pq 預填）
  • 約 2,936 筆

🤖 被動推薦（active=0，預設）

PagesController@fullTextPreview · 自動記錄
  • 使用者想看未收錄論文的全文 → 系統自動記錄此需求
  • 常缺少書目，需經 PQ 查詢補齊
  • 約 48,427 筆（大宗）
⚙️ 排程指令 ddc:pq-recommend 的查詢對象只限被動推薦（active=0）且尚無書目者——主動推薦的論文在填表時資料通常已知、不需再查。館員列表的「方式」欄即顯示 主動／被動。

3. ck_flag 與 valid 的變動

一句話分辨：ck_flag 記錄「為什麼／到哪一步」（多值字串、狀態機）； valid 記錄「能不能用」（0/1 結論）。valid 是結論，ck_flag 是理由——兩者在成功/失敗時會同時被寫入。

5. 容錯與韌性機制

  • 退避重試 ＋ IP↔帳號自動切換：初始化（登入/widget）失敗時，遞增退避時間後重新初始化，並在可用的 auth 模式間切換重試，連續失敗才放棄。
  • t:ac 空值偵測：ProQuest 改版導致路由 token 無法取得時，立即視為失敗並記錄，而非默默對每筆回報「查無」（這正是先前全站查詢失效的根因）。
  • 整批略過防護：單批次遇 VPN／DB 抖動或解析異常 → 跳過該批、留待重跑，不中斷整體流程。
  • DOI 正規化：查詢前去除尾端句點/空白（10746160. → 10746160）；垃圾型（掃描器探測）直接標記為 無效DOI，不浪費請求次數。
⚠️ recommends / recommends_on_pq 位於遠端 AWS（mysql_ddc_suggest），本機需經 VPN 連線；VPN 一斷即會出現 DB 連線逾時。正式長跑建議在伺服器內網執行最為穩定。

8. FAQ — 常見問題彙整

彙整本次開發與同仁提出的問題（ck_flag/valid、主動被動、batched、bought、登入語系、廢棄程式等）。

Q1．ck_flag 和 valid 差在哪？
ck_flag 記「為什麼／到哪一步」（多值狀態機字串）；valid 記「能不能用」（0/1 結論）。valid 是結論、ck_flag 是理由，兩者在成功／失敗時一起被寫。
Q2．剛新增 recommends 時 valid 是多少？
valid 預設 ＝ 1。不論主動、被動或 no_doi，一開始都是 valid=1；之後查無才會被改成 0。
Q3．主動推薦寫入的 ck_flag 是什麼？是 1 嗎？
不是。主動表單沒帶 ck_flag → 用欄位預設空字串 ''，valid=1。（實測全表空字串 ck_flag 全部是 active=1）
Q4．ck_flag='1' 和 '查詢成功' 差在哪？
兩者都代表「有書目」，但時間點不同：1 ＝被動推薦建立當下快取就命中（PagesController 即時帶入）；查詢成功 ＝排程事後把未解列補上（Phase 2 快取回填／Phase 3 PQ 查到）。
Q5．ck_flag=0 和空字串何時出現？
空字串 ＝主動推薦表單；0 ＝（已廢棄的）GetSuggestOnAws 從 AWS member 匯入時寫死的值。
Q6．排程指令的撈件條件？ck_flag／valid 有過濾嗎？
條件 ＝ active=0 且 bought=0 且 title 為空 且 ck_flag 不在 5 個終結狀態。ck_flag 有過濾（排除終結狀態）、valid 沒有過濾；真正關鍵是「title 還空著（尚未補書目）」。
Q7．哪些 ck_flag 新 batched 流程不會再產生？
查無此篇論文、尚未提供調閱、取得資料非原始指定之DOI（皆來自舊逐筆 search()／insertFromPq）。batched 查無一律記 查無相關結果。
Q8．使用者什麼時候進入推薦流程？
從 ProQuest 點某篇論文要看全文 → 連到 /doc/{doi}。若聯盟未收錄（Rawdata 查無）→ 自動記被動推薦並轉 OPAC 頁；使用者再點【我要全文】填表單 → 主動推薦。
Q9．主動 vs 被動推薦差在哪？
主動（active=1）＝使用者主動填推薦表單；被動（active=0，預設）＝想看未收錄全文時系統自動記錄。排程只處理被動的（主動的填表時資料已知）。
Q10．館員清單篩選的 ck_flag≠no_doi 是多餘的嗎？
不是。valid 預設=1，未解的 no_doi 也是 valid=1；只靠 valid=1 擋不掉它們（實測 714 列 no_doi+valid=1+bought=0），必須這條才能濾掉「無書目的未解需求」。
Q11．Batched 能查到哪些欄位？degree 為何留空？
結果列表頁就有 title／author／年／校／pq_id；degree／page／classification 只在詳細頁，為效率不點詳細頁（且 degree 已比照 ProQuest 從館員顯示移除）。
Q12．bought 怎麼讓論文從清單消失？
圖書館採購入庫（進 Rawdata）後，Batch::updateBought() 把該 DOI 的 recommends 設 bought=1 → 從清單消失，之後使用者可直接看全文。
Q13．「一次性重查以前查無」何時執行？
不在自動排程。手動執行 ddc:pq-recommend --recheck（把查無此篇／查無相關結果再撈出來重查，找到的救回、仍查無的不動）。
Q14．登入 ProQuest 有先切換語系嗎？切繁中嗎？
沒有切繁中。帳號登入會切英文（爬蟲解析依賴英文標記）；目前 IP 模式則不做任何切換。
Q15．GetSuggestOnAws（ddc:aws-suggest）還需要嗎？
不需要。它讀的 AWS member 表已不存在（遠端／本機都查無），跑了會報錯，屬死碼；建議連同 SuggestMember model 與 ddc_aws_suggest.bat 一併移除。推薦解析已統一於 ddc:pq-recommend。
